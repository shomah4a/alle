# ADR 0096: タイマーベースの定期再描画

## ステータス

承認

## コンテキスト

現在の再描画はキー入力後と非同期コマンド完了時の2パスのみで行われている。
非同期コマンド（shell-command等）のストリーミング出力のような中間出力は、次のキー入力または非同期コマンドの完了まで画面に反映されない。

理想的にはバッファ変更を自動検知して再描画する仕組み（EventNotifier等）が望ましいが、実装コストが高い。

## 決定

EditorThread のメインループで `keyQueue.take()`（無限ブロッキング）を `keyQueue.poll(200, TimeUnit.MILLISECONDS)`（タイムアウト付きポーリング）に変更する。

タイムアウト時（キー入力がない場合）にも `publishSnapshot()` を呼び出すことで、200ms間隔（5fps相当）の定期再描画を実現する。

タイムアウト値は `POLL_TIMEOUT_MS` として定数化する。

## 根拠

- TUIエディタのスナップショット生成とデルタ描画はゲームのフレームレンダリングと比べて極めて軽量であり、200ms間隔のポーリングによるオーバーヘッドは無視できる
- SnapshotExchanger は最新スナップショットのみ保持する設計であり、冗長なpublishが積み上がることはない
- 変更は EditorThread.run() 内の数行のみで、既存アーキテクチャへの影響が最小限

## 将来の発展

バッファ変更のダーティフラグやEventNotifierの導入により、変更がないときのポーリングを抑制する最適化が可能。ただし現時点では過剰最適化と判断し、シンプルなタイマー方式を採用する。
