# ADR 0075: TreeSitterStylerのスタイリング結果キャッシュ導入

## ステータス

承認

## コンテキスト

ADR 0074 で導入した TreeSitterStyler は、`styleDocument()` の呼び出しごとに
TSParser の生成・全文パース・クエリ実行・リソース破棄を行っている。
`RenderSnapshotFactory.create()` はレンダリングフレームごとにこのメソッドを呼ぶため、
テキスト変更がないスクロールやカーソル移動でもフルパースが発生し、不要な負荷がかかっている。

tree-sitter にはインクリメンタルパース（差分パース）の機能があるが、
`TSTree.edit(TSInputEdit)` によるツリーへの変更通知が必要であり、
バッファの変更イベントをスタイラーに伝達する仕組みの設計が伴う。

### 関連ADR

- ADR 0074: パーサーベースシンタックスハイライトの導入

## 決定

### ステップ1: テキスト未変更時のキャッシュ（本ADR）

TreeSitterStyler に前回のテキストとスタイリング結果をキャッシュする。

- `cachedText`: 前回の `styleDocument()` に渡されたフルテキスト
- `cachedResult`: 前回のスタイリング結果

`styleDocument()` 呼び出し時にテキストが前回と同一であればキャッシュ結果を返す。
テキストが異なれば従来通りフルパースを実行し、キャッシュを更新する。

TSTree 等のネイティブリソースはメソッドローカルで生成・破棄を継続するため、
リソースリークのリスクは発生しない。

### ステップ2: TSTreeキャッシュ + インクリメンタルパース（別タスク）

テキスト変更時の差分パースは以下の設計課題があり、別タスクとする。

- TSTree のネイティブリソースのライフサイクル管理（AutoCloseable対応）
- バッファ変更通知（TextChange → TSInputEdit 変換）の仕組み
- ドキュメント全体のUTF-8バイトオフセット計算

### スコープ

- 変更対象は `TreeSitterStyler.java` のみ
- `SyntaxStyler` インターフェースの変更は不要
- `RenderSnapshotFactory` の変更は不要
- スタイラーはバッファごとのモードインスタンスが保持しており、ステートのスコープはバッファに閉じている

## 根拠

- スクロール・カーソル移動・ウィンドウリサイズなど、テキスト変更を伴わないレンダリングが大半であり、キャッシュの効果が高い
- TreeSitterStyler の内部変更のみで完結し、外部インターフェースへの影響がない
- TSTree をフィールドに保持しないため、ネイティブリソース管理の問題を回避できる
