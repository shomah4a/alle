# ADR-0004: テキストモデルの設計

## ステータス

承認済み

## コンテキスト

エディタが内部でテキストデータを保持・操作するためのデータ構造を選定する必要がある。
候補としてGap Buffer、Piece Table、Ropeがある。
また、Unicode（サロゲートペア、絵文字等）を正しく扱う必要がある。

## 決定

### インターフェースと実装の分離

- `TextModel` インターフェースを `allei-core` に定義する
- テキストデータ構造の実装（Gap Buffer等）は `libs/` に汎用ライブラリとして配置する
- `allei-core` 側に `GapTextModel` を置き、`GapModel` をラップして `TextModel` を実装する

### コードポイント単位のAPI

- `TextModel` のインターフェースはコードポイント単位で操作する
- `GapModel`（libs）はchar単位の低レベル操作に徹する
- コードポイント↔charオフセットの変換は `GapTextModel` が担当する
- 書記素クラスタの扱いはカーソル移動等の上位層の責務とする

### 初期実装

- Gap Buffer を初期実装として採用する（Emacsと同方式）
- 将来的にPiece TableやRopeに差し替え可能

### 改行コード

- TextModel内部はLF(`\n`)に正規化する
- IO層で読み書き時に元の改行コード(CRLF, CR)との変換を行う

## 帰結

- インターフェース経由のアクセスにより、テキストデータ構造の差し替えが可能
- コードポイント単位のAPIにより、上位層がサロゲートペアを意識せずに済む
- Gap Bufferのシンプルさを活かしつつ、将来の拡張にも対応できる
