# ADR 0039: 入力・ロジック・描画の3スレッド分離

## ステータス

承認（改訂: 2スレッド → 3スレッド）

→ [ADR-0060](0060-actor-multithread.md) でスナップショットスレッドを追加し4スレッド構成に拡張。

## コンテキスト

ADR 0038 で描画スナップショットの分離が完了しているため、
スレッド分離の基盤が整った。

初期実装では入力スレッドとロジックスレッド（描画含む）の2分割としたが、
描画がロジックスレッドに含まれていたため、
スナップショット分離の設計意図（描画の分離）が活かされていなかった。

Lanterna はGUIフレームワークのようなイベントループを持たないため、
UIスレッドにLanterna操作を集約する必要はない。
readInput() と setCharacter()/refresh() は別スレッドから呼べる。

## 決定

### 3スレッド構成

- **入力スレッド（メインスレッド）**: キー入力の読み取りのみを担当
- **ロジックスレッド**: コマンド処理・スナップショット作成を担当
- **描画スレッド**: スナップショットの画面描画を担当

### スレッド間通信

- `LinkedBlockingQueue<KeyStroke>`: 入力スレッド → ロジックスレッド
- `AtomicReference<RenderSnapshot>` + wait/notify: ロジックスレッド → 描画スレッド

ロジックスレッドが高速に複数キーを処理した場合、
中間のスナップショットは不要であり最新のみを描画すればよいため、
キューではなくAtomicReferenceで最新値を上書きする。

### 入力スレッドのループ

1. `readKeyStroke()` でブロッキング待機
2. キーをキューに投入

### ロジックスレッドのループ

1. キューからキーを取り出し（ブロッキング）
2. `processKey()` でコマンド実行
3. `createSnapshot()` でスナップショット作成
4. `AtomicReference` にスナップショットを格納し、描画スレッドに通知

### 描画スレッドのループ

1. 新しいスナップショットの通知を待機
2. `AtomicReference` から最新スナップショットを取得
3. `renderSnapshot()` で画面描画
4. `refresh(DELTA)` で画面更新

### 終了処理

- `requestShutdown()` が呼ばれると `readKeyStroke()` が empty を返す
- 入力スレッドが POISON_PILL をキューに投入
- ロジックスレッドが POISON_PILL を検知して終了
- 入力スレッド（EditorRunner）がロジックスレッドのjoin後に描画スレッドに終了を通知
- 描画スレッドが終了通知を検知して終了

## 影響

- Frame/Window/Buffer へのアクセスはロジックスレッドに閉じる
- ScreenRenderer は描画スレッドが独占使用（同期不要）
- 入力スレッドはキーの読み取りとキュー投入のみ
- ロジックスレッドと描画スレッドはスナップショットを介して疎結合
